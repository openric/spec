# 2026-05-25 — Spec v0.38.1: RiC-AG cross-reference + RiC-CM Nav declaration

Doc-only patch release. Two open `drift-log.md` items resolved; no normative property mapping or profile claim changes.

## RiC-AG (Application Guidelines) — promoted to first-class

EGAD published [RiC-AG v0.1](https://ica-egad.github.io/RiC-AG/) in October 2025. It ships a crosswalk from the four legacy ICA standards (ISAD(G), ISAAR(CPF), ISDF, ISDIAH) to RiC-CM — the **exact source-standard set OpenRiC's `spec/mapping.md` covers**.

Until now RiC-AG was a passing one-line mention under the ICA-EGAD section of `related-implementations.md`. v0.38.1 promotes it:

- **`spec/mapping.md` §1.2 (new)** — declares RiC-AG as upstream-authoritative for the legacy-ICA-standards-to-RiC-CM crosswalk. OpenRiC defers to RiC-AG where the two agree; intentional divergences (the `openricx:` extension namespace minted where neither RiC-O 1.1 nor RiC-AG covers a concept) are documented and tracked as upstream-proposal candidates. The forthcoming EAC-CPF → RiC-O 1.1 mapping (also announced under RiC-AG) is flagged for cross-reference from §6.2 (Actor) when it lands.
- **`related-implementations.md`** — RiC-AG gets its own top-level section, sibling to Garance, RiC-O Converter, etc., with the explicit "why OpenRiC tracks it / how OpenRiC uses it / what OpenRiC does NOT do" structure used by the other entries.
- **`drift-log.md`** — records the promotion as a Closed item.

The framing matters: this is **not** a partnership claim or an endorsement claim. RiC-AG is EGAD's authoritative text; this spec defers on application-pattern questions and adds only what's needed for a deployable HTTP API surface (URI scheme, content negotiation, conformance profiles, write API, SPARQL access).

## RiC-CM navigator reconciliation — declared (option C)

The open question of how OpenRiC's `/reference/ric-cm/` browser relates to the Damigos / Ionian [RiC-CM Nav](https://dlib-ionian-university.github.io/ric-cm-nav/) was previously logged with three open options (deprecate / contribute / articulate). Resolved by **option C — articulate the distinction**:

- **OpenRiC's `/reference/ric-cm/`**: versioned, SPARQL-backed, declared-vs-inherited separation, tied to spec releases. A *spec-reference browser*.
- **Ionian RiC-CM Nav**: standalone single-page graph navigator for conceptual exploration. A *graph-exploration tool*.

The two address different reader needs (spec-version reference vs. conceptual traversal). The relationship is now formally declared as **complementary, not competing** on `related-implementations.md`. No deprecation, no fork; either tool can cite the other.

This declaration unblocks the **Damigos outreach** email (draft at `docs/outreach/damigos-second-implementation.md`) — the previous "currently undeclared" framing on the related-implementations page was a credibility risk for the pitch. Workbench notification reminder (B6) dropped to surface the outreach when ready to send.
