# 2026-05-25 — Spec v0.38.3: SPARQL probe fixes

Patch release shipped alongside OpenRiC service v0.10.0 (which adds the `sparql-access` profile claim). Two corrections to `conformance/probe.sh` and one new probe.

## Why this patch was needed

The OpenRiC reference service shipped a proper SPARQL Access implementation in v0.10.0 — canonical SPARQL Results JSON output, Accept-header content negotiation, SPARQL Update rejection, rate limiting, `/sparql/info` `void:Dataset` description. When the conformance probe was re-run against it, **the SPARQL probe failed** despite the endpoint being correct.

Two reasons:

### Reason 1 — Probe URL was being mangled by curl

The probe URL was:
```
/sparql?query=SELECT%20*%20WHERE%20{%20?s%20?p%20?o%20}%20LIMIT%201
```

The `{`, `}`, and `?` characters were not URL-encoded. `curl` (with default settings) treats `{...}` as a brace expansion glob and `?` as a query-string delimiter, so the URL was actually being transmitted in a mutilated form. Fuseki then parsed it as a malformed SPARQL query and returned `HTTP 400`.

The probe had been passing for months because the **previous OpenRiC implementation masked all backend errors as `HTTP 200`** with a wrapped JSON envelope (`{"sparql": "...", "results": {"error": ...}}`). The shape-predicate was a bare `.` (any non-null JSON), so the probe accepted the envelope as success. The new implementation, which correctly surfaces Fuseki's `400` as a 400 (per Core Discovery §4 RFC 7807 errors), revealed the probe bug.

**Fix:** pipe the SPARQL query through `jq -sRr @uri` before interpolating into the URL, same pattern the graph probe already uses for `SEED_URI`. Robust against curl globbing and against future query strings containing other reserved characters.

### Reason 2 — Probe predicate was too lax

The predicate `.` accepted any non-null JSON. That silently passed servers returning non-canonical envelopes around the result. **Fix:** assert `(.head.vars | type) == "array"`, which requires canonical SPARQL Results JSON per the SPARQL 1.1 Protocol — the only acceptable shape for a SELECT query response from a conformant server.

### And added: `/sparql/info` probe

The `sparql-access` profile §2.1 requires both `/sparql` and `/sparql/info`. Previous probe.sh tested only `/sparql`. Added the `/sparql/info` test as optional (matching the optional-status of the profile itself), checking for any non-null JSON response (since the endpoint may legitimately return Turtle or JSON-LD per content negotiation).

## Implication for other implementations

Any implementation that **claimed `sparql-access` while returning non-canonical SPARQL Results JSON** will now show up as non-conformant under the patched probe. This is the correct behavior — the previous probe was masking a real conformance bug. Implementations that emit canonical SPARQL Results JSON (which is the only thing SPARQL clients can parse anyway) pass.

The OpenRiC reference server (`ric.theahg.co.za`, service v0.10.0) passes 30/30 probes including both new SPARQL checks.
