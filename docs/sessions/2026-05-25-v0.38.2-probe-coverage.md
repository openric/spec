# 2026-05-25 — Spec v0.38.2: Conformance probe detail-endpoint coverage

Patch release. Closes a coverage gap surfaced by the v0.9.2 incident on the reference service.

## What was missing

`conformance/probe.sh` ran `probe_show_first` (i.e., extract first list item, GET its detail URL, assert HTTP 200 + valid `@id`) for only **3 of 8 entity types**: records, agents, repositories. The other five — places, rules, activities, instantiations, functions — were exercised only at the list level.

## Why it mattered

The reference-service v0.9.0 release (2026-04-25) shipped a regression: `serializeFunction` had a corrupted body (referenced undefined `$repo`/`$repositoryId`/`$ricRepo`) and `serializeRepository` didn't exist at all. `GET /functions/{id}` and `GET /repositories/{id}` both returned HTTP 500. The repositories case would have been caught by the existing probe; the functions case was invisible until 2026-05-25 because the probe never dereferenced a function. The bug lived for 35 days.

The audit memo on the reference-service side documents the masking pattern (`tempnam` errors from a missing `ProtectSystem` drop-in hiding the real exceptions) and the probe-coverage gap that compounded it. See [reference-service docs/sessions/2026-05-25-v0.9.1-and-v0.9.2.md](https://github.com/openric/service/blob/main/docs/sessions/2026-05-25-v0.9.1-and-v0.9.2.md).

## What this patch does

Adds 5 calls to the existing `probe_show_first` helper, slotted into the existing profile-gated sections:

```diff
 # ---- §2 Authority & Context — places, rules, activities ----
 if in_profile "authority-context"; then
   probe "list-places"     required GET "/places"     "$LIST_SHAPE"
+  probe_show_first "show-place"    "/places"     "/places"
   probe "list-rules"      required GET "/rules"      "$LIST_SHAPE"
+  probe_show_first "show-rule"     "/rules"      "/rules"
   probe "list-activities" required GET "/activities" "$LIST_SHAPE"
+  probe_show_first "show-activity" "/activities" "/activities"
   probe "places-flat"     optional GET "/places/flat" '.items | type == "array"'
 fi

 # ---- §3 Digital Object Linkage — instantiations, functions ----
 if in_profile "digital-object-linkage"; then
   probe "list-instantiations" required GET "/instantiations" "$LIST_SHAPE"
+  probe_show_first "show-instantiation" "/instantiations" "/instantiations"
   probe "list-functions"      optional GET "/functions"      "$LIST_SHAPE"
+  probe_show_first "show-function"      "/functions"      "/functions"
 fi
```

Probe surface goes **24 → 29** checks. Reference server passes all 29. The orphan-agent path (which previously failed because `listAgents` was leaking slug-less rows) now also passes thanks to the v0.9.2 `whereNotNull('slug.slug')` filter on the service side.

## Implication for other implementations

Any OpenRiC server that was previously passing the probe at 24/24 should still pass at 29/29 if its detail endpoints are correctly implemented. A new failure under this patch is **not** a probe artifact — it's a real server-side regression that was simply invisible before.

## What this patch does NOT do

- **No new spec prose.** No profile claim changes, no SHACL shape changes, no normative requirements added.
- **No `provenance-event` profile clarification.** The separate issue draft at `docs/issues/provenance-event-creator-unknown.md` is independent of this patch.
- **No `sparql-access` profile mount probe.** The probe already includes `/sparql?query=...` as optional in §4 (Graph Traversal); reference server happens to return 200 for the trivial query — investigation deferred to a future service-side task.
